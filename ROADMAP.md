# Roadmap: HR-ассистент

История решений — в git-истории репозитория, здесь её не дублируем.

## Текущее состояние (15.07.2026)

Архитектура v8, внутри контура Контура: Mattermost (бот `mmpy_bot`) → n8n (`n8n-common.testkontur.ru`) →
KonturGPT (диалог) → детерминированная генерация .docx (petrovich + JS) → Postgres
(`hr_disciplinary_assistant`). Подробности архитектуры и всех компонентов — в `CLAUDE.md`.

**Готово и отлажено вживую:**
- Mattermost вход/выход, Паспорт (OAuth), переезд на KonturGPT+контур.
- Идентификация участников через Staff (v4, Postgres-кэш staff_users).
- Промпт агента — полный цикл Шагов 1–7: приветствие → сценарий → идентификация → проверка
  сроков ст.193 → чек-лист обязательных действий → сбор недостающих фактов → подтверждение
  и генерация документов.
- Инструмент `scenario_checklist` — чек-лист по сценарию + запись ответов руководителя в БД
  (`hr_checklist_answers`), с жёстким правилом исключения пунктов, описывающих саму генерацию
  документов (акт/СЗ/подписание — это работа ассистента, не руководителя).
- Инструмент `Think` — самопроверка агента в 3 точках: соответствие режима работы сотрудника
  сценарию (после идентификации), противоречия в ответах чек-листа сценарию, полнота полей
  перед вызовом генерации.
- Генерация документов: мёртвая чек-лист-ветка вычищена из workflow; рендер акта/СЗ переписан
  на чистый JS (свой inflate + zip reader/writer через central directory) — `adm-zip`/`zlib`
  запрещены на инстансе, чтение и пересборка .docx работают без них, включая потоковый zip
  (Google Docs export); Postgres-схема и `alwaysOutputData` на фото-ноде починены.

**Написано, не проверено живым прогоном до конца:**
- Полный цикл Шаг 1→7 в реальном Mattermost-диалоге целиком (включая реальную генерацию
  и отправку файлов).

## Известные баги (в бэклоге, не чинили)

- **`role`/`query`/`source` всегда `null` в ответах «Идентификация пользователя».** Раньше
  между веткой «найден в кэше» и веткой «дозагружен из Стаффа» стояла Merge-нода («Слить ветки»),
  сохранявшая контекст (role/query) перед запросом профиля. Сейчас Merge-ноды нет — обе ветки
  сходятся напрямую в HTTP-ноду профиля, которая стирает входной JSON и заменяет его на голый
  ответ Стаффа. Нода «Профили: собрать ответ» при этом содержит устаревший комментарий «контекст
  берём из Слить ветки», но реально ссылается сама на себя (`$('Поиск данных по конкретным
  пользователям')`) — отсюда `role`/`query`/`source` всегда `null`.
  **Варианты фикса** (обсуждали, не выбрали): (а) вернуть явную Merge-ноду перед HTTP-запросом
  профиля — просто и надёжно; (б) переписать «Профили: собрать ответ», чтобы читать контекст из
  «Найден в кэше?» и «Отобрать нужных» напрямую, сопоставляя по `staff_id` — меньше нод, но больше
  риска нестыковок с $(...) в n8n при ветвлении.

## Бэклог (по приоритету)

1. **Починить `role`/`query`/`source` в идентификации** (см. баг выше) — влияет на то, что
   агент видит про «кто есть кто» среди участников.
2. **Доработка фактуры сбора фактов** — сейчас Шаг 6 спрашивает только время отсутствия,
   уважительную причину и описание доказательств. Не хватает: свидетелей должно быть до трёх
   (сейчас шаблон поддерживает только двух — нужно решить, расширять ли шаблон), время работы по
   трудовому договору, последствия отсутствия (для СЗ).
3. **Разделённые проверки сроков ст.193** — сейчас одна проверка на всё (`calc_deadlines`).
   По бизнес-логике акт и СЗ имеют разные требования к срокам (акт — день обнаружения, задним
   числом нельзя; СЗ — сроки привлечения). Сверить буфер (5 мес 15 дн / 20 дн) с Касмыниной О.
4. **Правки шаблонов документов** (Drive) — в акте нужно время вместо даты в конце, дерево
   отдела, обязательный город. В СЗ — разграничение «отсутствовал»/«отсутствует по настоящее
   время», раздел последствий.
5. **Приём фото-доказательств через Mattermost** (`/api/v4/files`) — сейчас отложено, механизм
   вставки скриншотов в .docx готов (см. «Рендер СЗ»), но приёма фото от руководителя в MM нет.
6. **`petrovich` не разрешён на инстансе** (`NODE_FUNCTION_ALLOW_EXTERNAL`) — сейчас работает
   JS-fallback склонения ФИО, но точность ниже, чем у библиотеки. Плюс `adm-zip` тем же запросом
   к инфре (хотя для рендера он больше не обязателен — заменён своим кодом).
7. **Реальный RAG по ЛНА/ИД** — чек-лист сейчас грузится из Drive целиком, без поиска. Заглушка
   в «Генерация документов» (`disabled`-островок) под это уже есть.
8. **Vision (анализ доказательств)** — сейчас заглушка, просто передаёт текстовое описание.
9. **Приоритизация новых сценариев** — сейчас поддержан только `progul_ochny`. Кандидаты:
   дистанционное отсутствие, опоздание, НДО (по итогам обсуждений с заказчиком).
