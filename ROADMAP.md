# Roadmap: HR-ассистент — план доработок

## Статус MVP
Реализован сценарий "Прогул офисник": диалог → проверка сроков → акт + СЗ.
Работает локально в n8n с Ollama qwen2.5:7b.

---

## Открытые задачи

### 🔴 P0 — Критично (блокирует нормальное использование)

#### 1. Нет нормального интерфейса + нет промежуточных ответов
**Проблема:** Встроенный n8n чат — не для пользователей. Запрос висит молча
пока цепочка (3× Ollama вызовов) не завершится — это 1-3 минуты тишины.

**Что нужно:**
- Перевести на Telegram Bot (n8n Telegram Trigger + Send Message)
- В каждом шаге цепочки — промежуточное сообщение в чат:
  `"⏳ Анализирую нарушение..."` → `"⏳ Составляю акт..."` → `"⏳ Пишу служебную..."` → результат
- Либо Mattermost если Telegram не подходит

**Зависит от:** наличия Telegram бота (токен)

---

#### 2. LLM ломается в нетиповых сценариях
**Проблема:** qwen2.5:7b путается при отклонении от happy path —
неполные данные, нестандартные ситуации, граничные случаи.

**Что нужно выяснить:** проблема в модели или в промптах?
- Проверить промпты на robustness: что происходит если нет имени, нет даты и т.д.
- Добавить explicit fallback-инструкции в промпты ("если данных нет — пиши [УТОЧНИТЬ]")
- Рассмотреть смену модели: qwen2.5:14b / mistral / другая локальная

---

### 🟡 P1 — Важно (ограничивает функциональность)

#### 3. Нет гибкой подгрузки шаблонов + нет рендера docx
**Проблема:** Шаблоны вшиты в код (fs недоступен в n8n). Результат — markdown-текст,
а не .docx который можно распечатать и подписать.

**Что нужно:**
- Шаблоны хранить в PostgreSQL или Google Drive / S3
- Рендер финального документа в .docx (библиотека docxtemplater или аналог)
- Отправлять файл пользователю в Telegram (Send Document)

---

#### 4. Нет RAG по ЛНА и ИД (локальные нормативные акты, инструкции)
**Проблема:** Сейчас чеклист и ТК захардкожены. При росте числа НПА
(ПВТР, ЛНА, трудовые договоры) — неуправляемо.

**Что нужно:**
- Векторная БД: Qdrant (уже планировалось) или pgvector в PostgreSQL
- Embedding: Ollama nomic-embed-text (уже установлен)
- Загрузка документов: PDF/docx → chunking → embeddings → Qdrant
- RAG-поиск: по запросу → топ-N чанков → в промпт вместо полного текста
- Интеграция с RAG-AAS ЦИИ (Вова Поздняков, июль 2026)

---

#### 5. Нет прочтения артефактов (зрение + валидация)
**Проблема:** Vision-заглушка принимает только текстовое описание доказательств.
Руководитель не может прикрепить скрин СКУД / переписку.

**Что нужно:**
- Принимать изображения через Telegram (photo message)
- Ollama vision-модель: qwen2-vl или llava (запустить через `ollama pull`)
- HTTP Request к http://127.0.0.1:11434/api/generate с base64 images[]
- Валидация: модель описывает что видит → включается в промпт

---

#### 6. Нет записи данных в PostgreSQL (аналитика)
**Проблема:** Каждое обращение обрабатывается и забывается. Нет истории,
нет аналитики по нарушениям, нельзя отследить паттерны.

**Что нужно:**
- Таблица `hr_cases`: id, scenario, employee_name, department,
  violation_date, manager_name, created_at, documents_generated
- Запись после успешной генерации документов
- По запросу — запись дополнительных полей (решение, исход дела)

---

### 🟢 P2 — Инфраструктура

#### 7. Всё крутится локально
**Проблема:** n8n и Ollama на MacBook Олега. Недоступно другим,
нет отказоустойчивости, зависит от железа.

**Варианты:**
- VPS + Docker Compose (n8n + Ollama) — минимальный деплой
- Ollama с GPU-сервером Контура (если есть)
- n8n Cloud + Ollama API (если допустимо по данным)
- Поэтапно: сначала n8n в облако, Ollama пока локально через ngrok/tunnel

---

## Порядок работы

```
P0.1 Telegram Bot          → разблокирует нормальное тестирование
P0.2 Промпты robustness    → параллельно с Telegram
P1.1 Рендер docx           → после Telegram (чтобы отправлять файл)
P1.2 RAG + Qdrant          → после стабилизации промптов
P1.3 Vision                → после RAG
P1.4 PostgreSQL аналитика  → можно в любой момент (независимо)
P2   Деплой на сервер      → после всего P1
```
