# HR-консультант — CLAUDE.md

## Что это

ИИ-ассистент для руководителей Контура по оформлению дисциплинарных документов (акт об отсутствии,
служебная записка). Стек: **n8n** + **KonturGPT (диалог)** + **детерминированная генерация .docx**
+ **Postgres** + **Google Drive (шаблоны)**. Статус: **архитектура v8** — переезд **внутрь контура
Контура**: интерфейс **Mattermost** (бот на `mmpy_bot`), диалог на **KonturGPT**, идентификация
сотрудников через **Staff API** (токен из Контур Паспорта), генерация без LLM (petrovich + JS).

> 🚦 **Статус (15.07.2026):** промпт агента реализует полный цикл **Шагов 1–7** — приветствие →
> определение сценария → идентификация участников (Staff) → проверка сроков ст.193 →
> чек-лист обязательных действий (`scenario_checklist`) → сбор недостающих фактов → подтверждение
> и вызов генерации документов (`generate_documents`). Инструмент `Think` — самопроверка агента
> в 3 точках (соответствие сценария режиму работы сотрудника, противоречия в чек-листе, полнота
> полей перед генерацией).
> Рендер `.docx` переписан на чистый JS (свой inflate + zip reader/writer через central
> directory) — `adm-zip`/`zlib` запрещены на инстансе n8n; читает и потоковый zip (так экспортирует
> Google Docs). Не пройден живым прогоном: полный цикл Шаг 1→7 в реальном Mattermost-диалоге
> целиком, включая саму генерацию. Известный баг и весь бэклог — в `ROADMAP.md`.

> Теперь диалог на **KonturGPT** (внутренняя модель) → геоблок не мешает, n8n живёт внутри контура,
> где доступны Mattermost и Staff.

> **Почему Mattermost, а не Telegram:** заказчик (кадровый блок) утвердил единственный канал —
> Mattermost (внутренний; TG отвергли по данным). У MM нет n8n-триггера, свой Node-WS рвался (1006),
> поэтому вход — бот на **`mmpy_bot`** (Python, `bot/`), слушает WebSocket → POST в n8n webhook.

> **Почему генерация без LLM:** qwen копировала вход/ошибалась в склонении ФИО. Заменено на
> детерминированный код (petrovich + JS) — быстрее, предсказуемо, без GPU.

## Архитектура v8 — три workflow

```
Mattermost (личка боту)
    ↓  бот на mmpy_bot (bot/) слушает WebSocket, POST в webhook
HR-агент.json → Webhook (/mattermost-in)
    ↓
Нормализация входа (Code: body.channel_id/user_id/message → chat_id = channel_id)
    ↓  [фото через MM — отложено; телеграм-ветка отключена]
HR-ассистент (AI Agent, KonturGPT preview-pro)
    ├── Postgres память (Postgres Chat Memory, sessionKey = channel_id)
    ├── Tool: calc_deadlines (toolCode) — относительные даты + сроки ст.193
    ├── Tool: Идентификация пользователя → Идентификация пользователя.json (Staff)
    ├── Tool: scenario_checklist → Чеклист сценария.json (чек-лист + запись ответов)
    ├── Tool: Think (toolThink) — самопроверка (см. промпт, 3 точки)
    └── Tool: generate_documents → Генерация документов.json
    ↓
Формат ответа (Code: markdown как есть + разбивка ≤15000)
    ↓
Отправить сообщение (HTTP POST /api/v4/posts, channel_id + text)

Идентификация пользователя.json (tool, v4 — ✅ отлажена вживую):
    Trigger(passthrough: subject/manager/witness1..3 — email/логин, любое подмножество)
    → Получение токена (Паспорт) → PG SELECT кэша staff_users (alwaysOutputData)
    → Поиск нужных по массиву (Code: item на человека, матч email/localpart/login/staff_id)
    → IF Найден в кэше? (per-item)
        TRUE  → Слить ветки (Merge append)
        FALSE → Собрать отсутствующих (1 item) → Стафф: ВСЕ (view=light, пагинация)
                → пакет для SQL → PG батч-upsert (jsonb_to_recordset, 1 запрос)
                → Отобрать нужных → Слить ветки
    → профиль /api/Users/{id} (по каждому) → Профили: собрать ответ. Выход — массив, item на НАЙДЕННОГО:
      { role, query, found, staff_id, fio, surname, firstname, patronymic, post, department,
        organization, city, office, email, login, status, source }
      Кого не нашли — в ответе нет (= not_found); сбой профиля → found:false + error "staff_error: ...".
    ⚠️ Кэш — PG-таблица staff_users («PG: init staff_users» дёрнуть раз). view=large НЕ содержит
    id/email на верхнем уровне — для кэша только light.
    ⚠️ БАГ (в бэклоге): role/query/source в финальном ответе всегда null — раньше контекст перед
    запросом профиля сохраняла Merge-нода «Слить ветки», сейчас её нет. Подробности — ROADMAP.

Чеклист сценария.json (tool, scenario_checklist):
    Trigger(scenario, chat_id, answers?)
    → без answers: Switch по scenario → Drive: скачать чеклист → парсинг → { checklist_content,
      нужен_акт, нужна_сз }
    → с answers ([{item, done, comment}]): Postgres батч-upsert в hr_checklist_answers → { saved, count }

Инициализация схемы Postgres: см. db/schema.sql (применить вручную — psql или разовый скрипт,
отдельного workflow в репозитории больше нет).
```

**Поток диалога (текущий, промпт реализует полностью, Шаги 1–7):**
1. Приветствие + что умею.
2. Определение сценария (не поддержан → к экспертам, не угадывать).
3. Идентификация участников через Staff (email/логин, один вызов тула на пакет, можно повторять) →
   подтверждение у руководителя → Think (сценарий соответствует режиму работы сотрудника?).
4. Проверка сроков ст.193 (`calc_deadlines`) — истёк → к экспертам, стоп; не истёк → дальше.
5. Чек-лист обязательных действий (`scenario_checklist`) — получить, отфильтровать пункты про саму
   генерацию (жёсткое правило-исключение), задать оставшиеся, записать ответы → Think (нет ли в
   ответах противоречия сценарию).
6. Сбор недостающих фактов — только то, что ещё не известно: время отсутствия, уважительная
   причина, доказательства. Адрес/город и попытки связи уже есть (из Стаффа и чек-листа).
7. Think (полнота полей) → сводка → подтверждение → `generate_documents`.

**Что уже известно к Шагу 7 (не переспрашивать):** ФИО/должность/подразделение сотрудника,
руководителя, свидетелей (Стафф, маппинг `fio`→`name`, `post`→`position`); город/адрес рабочего
места сотрудника (Стафф); попытки связи (ответ по чек-листу, Шаг 5); даты нарушения/обнаружения
(`calc_deadlines`, Шаг 4). Приём фото-доказательств через Mattermost — отложено (ROADMAP).

**Передача в генерацию (`generate_documents`):** `scenario`, `employee`, `manager`, `workplace`,
`witnesses[2]`, `violation`, `respectableReason(+Comment)`, `evidenceDescription`, `contactAttempts`,
`evidenceFiles[]`, `chat_id`.

**Проверка сроков (calc_deadlines) — Шаг 4 промпта.** Агент спрашивает у руководителя дату
нарушения и дату обнаружения свободным текстом (ГГГГ-ММ-ДД, ДД.ММ.ГГГГ, «вчера/сегодня/завтра»,
дни недели — «в прошлый/позапрошлый вторник», «на этой/прошлой неделе в среду», голый день недели =
ближайшее прошлое вхождение, — и недели как длительность — «неделю назад», «через N недель») и
передаёт как есть в Code-нода-тул строкой `"violationDate=...; discoveryDate=..."` → JSON
`{ isExpired, expiredReason, violationDeadline, discoveryDeadline }`. Агент НЕ считает и НЕ
парсит даты сам — это делает инструмент. Срок истёк → генерация не запускается, направить к
Касмыниной О. / Черепановой Т. **Сроки ст.193 — ТОЛЬКО здесь.**

> ⚠️ **Не вкладывать саб-агентов!** Вложенный AI Agent (`agentTool`) в n8n падает с
> «The Tool attempted to return an engine request, which is not supported in Agents». Логику —
> на `toolCode`/`toolWorkflow`/HTTP. См. memory `n8n-nested-agent-engine-request`.

> **Буферные сроки** (внутренний буфер Контура, НЕ дословно ТК; baseline ТК — 6 мес / 30 дн):
> от нарушения **+5 мес. 15 дн.**, от обнаружения **+20 дн.** Зашиты в `calc_deadlines`. Уточнять у Касмыниной О.

**Генерация документов.json — конвейер (без LLM):**

```
Вызов из агента (Execute Workflow Trigger)
  → Нормализовать данные (Code: распаковка query, chat_id, scenario, evidenceFiles)
  → Vision: анализ доказательств (заглушка)
  → Значения документов (Code: context → 25 плейсхолдеров + DOC_MAP флаги)
  → Объединить значения (Code: petrovich-склонение ФИО + обстоятельства + приложения)
  → Статус: готовлю документы ⏳ (Mattermost POST /api/v4/posts)
  → Нужен акт? (IF) → [Drive: скачать акт → Рендер акта → MM: загрузить акт (/api/v4/files) → MM: пост акт]
  → Нужна СЗ? (IF) → [Postgres: фото кейса → Drive: скачать СЗ
                      → Рендер СЗ → MM: загрузить СЗ → MM: пост СЗ]
  → Финальный ответ → Postgres: запись кейса (hr_cases) → Вернуть результат

  ╎ ВЫКЛЮЧЕННЫЙ островок-задел под RAG (disabled, в обход):
  └─ Сборка промпта (нарушение) → Ollama: нарушения → Объединить нарушения
```

> 🔮 **Островок нарушений выключен** (`disabled`) — это задел под будущий RAG. Раньше первый
> Ollama-вызов дублировал «обстоятельства» с кривой ссылкой на ТК и шёл в мусорку. Включить при
> подключении векторной базы ЛНА (ROADMAP P1.2). Поток идёт в обход: Vision → Значения документов.

**Генерация полей (нода «Объединить значения», без LLM):**
- `фио_работника_дат` — склонение ФИО в **дательный** падеж («применить к кому») через **`petrovich`**;
  если petrovich недоступен — встроенный JS-fallback (типовые русские ФИО). Заполняется и legacy-ключ
  `фио_работника_род` (для совместимости, пока шаблон в Drive не переименован).
- `обстоятельства` — склейка `contactAttempts` + уважительной причины.
- `приложения` — нумерованный список из `evidenceDescription`.

**Рендер .docx (свой JS, без adm-zip):**
- Шаблоны — `.docx` в **Google Drive** с цельными `{{плейсхолдерами}}`. Команда правит их в Drive.
- **Service Account** `N8N-kontur` **только скачивает** (copy/create нельзя — 403 storageQuotaExceeded,
  нет квоты). Заполнение локально, в Drive ничего не пишется.
- `adm-zip` и `zlib` **запрещены** на инстансе n8n (`Module '...' is disallowed`) — рендер написан
  на чистом JS без единого `require`: свой DEFLATE-инфлейт (RFC 1951) + ZIP reader/writer.
  Непереписанные записи копируются как есть (byte-for-byte, без пересжатия); изменённые
  (`word/document.xml`, для СЗ ещё `_rels`/`[Content_Types].xml`) перезаписываются STORED
  (без сжатия) — поэтому DEFLATE-энкодер не нужен, только декодер.
- ⚠️ **Читать zip нужно через central directory, не последовательным обходом локальных
  заголовков.** Google Docs при экспорте в .docx использует потоковый режим (general purpose
  bit 3 — размеры в локальном заголовке нулевые, реальные размеры в central directory в конце
  архива). Код в «Рендер акта»/«Рендер СЗ» это учитывает.
- **Хардкод в «Значения документов»:** организация `АО «ПФ «СКБ Контур»`; адресат СЗ `Директору`
  + пустой ФИО руководителя; `время_составления` акта — авто.

> `petrovich` (склонение ФИО) всё ещё не разрешён на инстансе — работает JS-fallback. См. ROADMAP.

**Скриншоты-доказательства в СЗ (механика готова, приём фото в MM отложен):**
- ⏸ В v8 (Mattermost) **приём фото отложен** (телеграм-ветка отключена). Ниже — как это работало
  на Telegram и остаётся в `Рендер СЗ` под будущее подключение фото через MM (`/api/v4/files`).
- Фото → base64 → `hr_evidence.file_b64`. При генерации СЗ `Postgres: фото кейса` достаёт байты по
  `chat_id`; `Рендер СЗ` вставляет каждое фото в .docx через **OOXML** (`word/media/`, relationship,
  `<w:drawing>`), подпись «Скриншот N», якорь `{{скриншоты}}`.
- ⚠️ **Расширение media-файла должно быть в `[Content_Types].xml`** — иначе Word «документ повреждён».
  Сохраняем как `.jpeg` + нода регистрирует расширение при вставке.

**Вариативность документов (DOC_MAP).** В «Значения документов» карта
`DOC_MAP = { scenario: { нужен_акт, нужна_сз } }`. Развилки `Нужен акт?` / `Нужна СЗ?` включают
рендер по флагам.

> ⚠️ **`.first()`, а не `.item`** в far-ссылках после Telegram/HTTP/Drive-нод (они роняют вход).
> `.item` рвётся при разрыве потока и поштучном Execute → развилки в False, `chat_id is empty`,
> поля undefined. **Отлаживать только полным прогоном** (через бота). См. memory `n8n-first-vs-item-far-refs`.

> ⚠️ **Postgres + chat_id:** колонка `chat_id` — `TEXT`, а из n8n приходит **числом**. В SQL
> сравнение `chat_id = $1` падает («operator does not exist: text = integer»). Использовать
> `chat_id = $1::text` (нода «Postgres: фото кейса»).

### Среда (v8): n8n внутри контура Контура
n8n живёт на `n8n-common.testkontur.ru` (общий инстанс). Диалог — KonturGPT (внутри контура),
Mattermost/Staff/Паспорт доступны напрямую. Внешний деплой (Нидерланды) больше не нужен.

### Mattermost-бот (вход) — локально или в контейнере
```bash
cd bot
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
MM_BOT_TOKEN='...' N8N_WEBHOOK_URL='https://n8n-common.testkontur.ru/webhook/mattermost-in' \
  MM_SSL_VERIFY=false python bot.py
```
> ⚠️ Python 3.14: нужен ручной `asyncio.new_event_loop()` до `bot.run()` (уже в `bot.py`).
> Бот ловит ЛС (`direct_only`) → POST в n8n. Ответ постит сам n8n (нода `/api/v4/posts`).

### Credentials в n8n (UI)
- **KonturGPT** (`Kontur.GPT`, openAiApi) — внутренняя модель, `preview-pro`.
- **Контур. Mattermost** (httpBearerAuth) — токен бота. Для входящих/исходящих `/api/v4/*`.
- **Контур. Паспорт** (httpHeaderAuth) — `Authorization: Basic base64(clientId:clientSecret)`; на ноде «Получение токена».
- **n8n_hr_disciplinary_assistant** (Postgres) — host `devof-pt-vxsa1.dev.kontur.ru`, db `n8n`,
  схема `hr_disciplinary_assistant`, **SSL = disable** (сервер не отдаёт TLS). 🔴 нужны права от DBA.
- **Google Drive (Service Account)** `N8N-kontur` — только чтение; шаблоны/чеклист расшарить на e-mail SA.

### Импорт workflow
1. Импортировать все 4: `HR-агент`, `Генерация документов`, `Идентификация пользователя`, `Чеклист сценария`.
2. Проверить/перепривязать credentials (при импорте отвязываются): KonturGPT, Mattermost, Паспорт, Postgres, Drive.
3. Перепривязать `workflowId` у tools в HR-агенте: `generate_documents` → «Генерация документов»,
   `Идентификация пользователя` → её workflow, `scenario_checklist` → «Чеклист сценария».
4. Применить `db/schema.sql` вручную (psql), если схема/таблицы ещё не созданы.
5. Активировать `HR-агент` (webhook `/mattermost-in`).

> `petrovich` (склонение ФИО) не разрешён на инстансе — работает JS-fallback. Рендер `.docx`
> `adm-zip`/`zlib` не требует (свой код, см. выше).

## Как добавить новый сценарий

1. `knowledge/{scenario}_checklist.md` → выгрузить в Drive, прописать fileId в ноду «Drive: скачать
   чеклист» (в `Чеклист сценария.json`) и `DOC_MAP` там же (`нужен_акт`/`нужна_сз`).
2. `.docx`-шаблоны с `{{плейсхолдерами}}` в Drive, fileId в «Значения документов» (в `Генерация
   документов.json`) + строка в тамошней `DOC_MAP` (дублируется намеренно, см. ROADMAP).
3. В системном промпте агента (Шаг 2) — когда присваивать новый `scenario`.

## Ключевые ограничения ассистента

- Никогда ничего не додумывает — только факты от руководителя.
- Граница: до передачи пакета документов эксперту включительно.
- Не под сценарий → к Касмыниной О. / Черепановой Т. ИБ/КТ → юр.служба / ОИБО.
- Не цитирует статьи ТК по памяти; сроки только из `calc_deadlines`.

## Роли

| Роль | Описание |
|------|----------|
| Руководитель | Инициирует, предоставляет факты |
| ИИ-ассистент | Ведёт диалог, формирует документы |
