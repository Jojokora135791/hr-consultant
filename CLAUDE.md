# HR-консультант — CLAUDE.md

## Что это

ИИ-ассистент для руководителей Контура по оформлению дисциплинарных документов (акт об отсутствии, служебная записка). Реализуется на **n8n** + **Claude (диалог)** + **Ollama (генерация)**. Статус: активная разработка, **архитектура v5 (два workflow, гибрид облако/локаль)**.

## Архитектура v5 — два workflow

Вся система собрана в **два workflow**: диалоговый агент (Claude) ведёт диалог, проверяет сроки и вызывает генерацию (Ollama).

```
HR-агент.json (точка входа)
    ↓
Чат с руководителем (Chat Trigger)
    ↓
HR-ассистент (AI Agent, Claude Opus 4.8 через Anthropic Chat Model)
    ├── Память диалога (Window Buffer, 30 сообщений)
    ├── Tool: calc_deadlines («Рассчитать сроки ст.», toolCode/JS) — ПРЯМОЙ тул
    │         → детерминированный расчёт дедлайнов ст.193, без LLM-арифметики
    └── Tool: generate_documents → Генерация документов.json (toolWorkflow)
```

**Поток диалога (системный промпт агента):** приветствие → сбор фактов по одному вопросу →
суммаризация → **Шаг 4: проверка сроков** (обязательно ДО генерации) → генерация → выдача.

**Проверка сроков (Шаг 4).** Агент сам вызывает инструмент `calc_deadlines` (Code-нода на JS),
передавая строку `"violationDate=ГГГГ-ММ-ДД; discoveryDate=ГГГГ-ММ-ДД"`. Тот возвращает JSON:
`{ isExpired, expiredReason, violationDeadline, discoveryDeadline }`. Агент НЕ считает даты сам —
доверяет только числам из тула. Если срок истёк — генерация не запускается, руководителя направляют
к Касмыниной О. / Черепановой Т.

> ⚠️ **Не вкладывать саб-агентов!** В v4 проверку сроков делал отдельный AI Agent (`agentTool`
> «Ресерчёр по срокам»). Вложенный агент в n8n падает с ошибкой *«The Tool attempted to return an
> engine request, which is not supported in Agents»*. В v5 он убран, `calc_deadlines` висит прямым
> тулом. Логику саб-агентов всегда переносить на `toolCode`/`toolWorkflow`/HTTP Request Tool.

> **Буферные сроки** (внутренний буфер Контура, НЕ дословно ТК РФ; baseline ТК — 6 мес / 30 дн):
> - от даты нарушения: **+5 мес. 15 дн.**
> - от даты обнаружения: **+20 дн.**
> Зашиты прямо в JS-код `calc_deadlines`. Уточнять у Касмыниной О.

**Генерация документов.json — конвейер генерации (только генерация, без проверки сроков, на Ollama):**

```
Вызов из агента (Execute Workflow Trigger)
  → Нормализовать данные (Code)
  → Выбор сценария (Switch по scenario)
        ├ progul_ochny → Прочитать данные прогула
        └ zaglushka    → Сценарий: заглушка
             ↓
        Vision: анализ доказательств (заглушка, Code)
        → Промпт: нарушения (RAG, Code) → Ollama: нарушения (HTTP)
        → Объединить нарушения (Code)
        → Нужен акт? (Switch)
              ├ да  → Промпт акта → Ollama: генерация акта (HTTP) → Объединить акт
              └ нет → «Акт не нужен» (Code) → Объединить акт
        → Промпт: служебная записка → Ollama: генерация СЗ (HTTP)
        → Финальный ответ (Code)
```

**Принцип "всё в Code-нодах":** чеклисты и шаблоны **вшиты прямо в Code-ноды** «Генерации документов».
В n8n нет доступа к файловой системе (`fs.readFileSync` не работает), поэтому контент захардкожен.
Файлы `knowledge/*.md` и `templates/*.md` остаются в репозитории как **исходник правды** — при
изменении их нужно вручную перенести в соответствующие Code-ноды.

**Вызовы Ollama внутри генерации** идут через HTTP Request к `http://127.0.0.1:11434/api/generate`
(не через langchain-ноду — langchain-модель используется только в диалоговом агенте, и там это Claude).

**Маршрутизация сценариев:** агент в системном промпте присваивает `scenario = "progul_ochny"` только
для очного прогула (сотрудник не ходит в офис), во всех остальных случаях — `"zaglushka"`.

## Настройка и запуск

### Запуск окружения (вручную)
```bash
brew services start ollama          # обычно уже авто-старт (нужна для генерации)
n8n start                           # http://localhost:5678
```

### 1. Credentials в n8n

**Anthropic API** (для диалогового агента) — n8n → Settings → Credentials → Add → Anthropic
- API Key: `sk-ant-...` (создать в console.anthropic.com → API Keys; нужен пополненный баланс)
- Base URL: `https://api.anthropic.com` (ровно так, без `/v1`)

**Ollama API** (для генерации документов) — n8n → Settings → Credentials → Add → Ollama API
- Base URL: `http://127.0.0.1:11434`
> ⚠️ Именно `127.0.0.1`, не `localhost` — macOS резолвит в IPv6 (::1), Ollama слушает только IPv4

### 2. Импортировать workflow (вручную через UI n8n)
1. Импортировать `Генерация документов.json` (Import from File)
2. Импортировать `HR-агент.json` → проверить credential Anthropic на ноде `Anthropic Chat Model`
3. В ноде `Генерация документов` (HR-агент) убедиться, что `workflowId` указывает на импортированную «Генерацию документов»

> Workflow подгружаются в n8n вручную через интерфейс. Скрипта автозапуска нет.
> Credential PostgreSQL для текущей архитектуры **не требуется** (сроки считаются в JS).

### 3. Тест
Открой `HR-агент` → Test Workflow → в чате: `"Помоги оформить прогул"`

## Как добавить новый сценарий

1. Создать `knowledge/{scenario}_checklist.md` — чеклист из соответствующего .docx
2. Создать `templates/sz_{scenario}.md` / `act_{scenario}.md` — если нужны другие шаблоны
3. В `Генерация документов.json`:
   - добавить ветку в ноду `Выбор сценария` (Switch) под новый `scenario`-ключ
   - вшить чеклист/шаблон в соответствующие Code-ноды (промпты)
4. В системном промпте агента (`HR-агент.json`) описать, когда присваивать новый `scenario`-ключ

## Ключевые ограничения ассистента

- Никогда ничего не додумывает — только факты от руководителя
- Граница: до передачи пакета документов эксперту включительно
- Если ситуация не под сценарий → направить к Касмыниной О. или Черепановой Т.
- Если нарушение связано с ИБ / коммерческой тайной → юр. служба / ОИБО
- Не цитирует статьи ТК по памяти; сроки берёт только из `calc_deadlines`, не считает даты в уме

## Роли в процессе

| Роль | Описание |
|------|----------|
| Руководитель | Пользователь ассистента. Инициирует, предоставляет факты |
| ИИ-ассистент | Ведёт диалог, консультирует, формирует документы |
| Эксперт кадрового учёта | Черепанова Т., Касмынина О. Принимает финальный пакет |
| ОКУ | Получает табельные данные (НН), приказы |
