# HR-консультант — CLAUDE.md

## Что это

ИИ-ассистент для руководителей Контура по оформлению дисциплинарных документов (акт об отсутствии, служебная записка). Реализуется на **n8n** + **Ollama** (локально). Статус: активная разработка, **архитектура v4 (два workflow)**.

## Архитектура v4 — два workflow

Вся система собрана в **два workflow**: основной агент ведёт диалог, проверяет сроки и вызывает генерацию.

```
hr_agent.json (HR-агент, точка входа, workflowId=xvn4xaB9ZexlCVDG)
    ↓
Чат с руководителем (Chat Trigger)
    ↓
HR-ассистент (AI Agent, qwen2.5:7b, temperature 0.3)
    ├── Память диалога (Window Buffer, 30 сообщений)
    ├── Tool: «Ресерчёр по срокам» (саб-агент проверки сроков ст.193)
    │        ├── свой Ollama Chat Model + Simple Memory (20)
    │        └── Tool: calc_deadlines («Рассчитать сроки ст.193», toolCode/JS)
    │                  → детерминированный расчёт дедлайнов, без LLM-арифметики
    └── Tool: generate_documents → hr_generate.json (workflowId=VSXqMIrZYbAFUGYk)
```

**Поток диалога (системный промпт агента):** приветствие → сбор фактов по одному вопросу →
суммаризация → **Шаг 4: проверка сроков** (обязательно ДО генерации) → генерация → выдача.

**Проверка сроков (Шаг 4).** Агент вызывает саб-агента «Ресерчёр по срокам», передавая строку
`"violationDate=ГГГГ-ММ-ДД; discoveryDate=ГГГГ-ММ-ДД"`. Саб-агент НЕ считает даты сам — он обязан
вызвать инструмент `calc_deadlines` (Code-нода на JS). Тот возвращает JSON:
`{ isExpired, expiredReason, violationDeadline, discoveryDeadline }`. Если срок истёк — генерация
не запускается, руководителя направляют к Касмыниной О. / Черепановой Т.

> **Буферные сроки** (внутренний буфер Контура, НЕ дословно ТК РФ; baseline ТК — 6 мес / 30 дн):
> - от даты нарушения: **+5 мес. 15 дн.**
> - от даты обнаружения: **+20 дн.**
> Зашиты прямо в JS-код `calc_deadlines`. Уточнять у Касмыниной О.

**hr_generate.json — конвейер генерации (только генерация, без проверки сроков):**

```
Вызов из агента (Execute Workflow Trigger)
  → Нормализовать данные (Code)
  → Выбор сценария (Switch по scenario)
        ├ progul_ochny → Прочитать данные прогула
        └ zaglushka    → Сценарий: заглушка
             ↓
        Vision: анализ доказательств (заглушка, Code)
        → Промпт: нарушения (RAG, Code) → Ollama: нарушения (HTTP)
        → Объединить нарушения (Code)
        → Нужен акт? (Switch)
              ├ да  → Промпт акта → Ollama: генерация акта (HTTP) → Объединить акт
              └ нет → «Акт не нужен» (Code) → Объединить акт
        → Промпт: служебная записка → Ollama: генерация СЗ (HTTP)
        → Финальный ответ (Code)
```

**Принцип "всё в Code-нодах":** чеклисты и шаблоны **вшиты прямо в Code-ноды** hr_generate.
В n8n нет доступа к файловой системе (`fs.readFileSync` не работает), поэтому контент захардкожен.
Файлы `knowledge/*.md` и `templates/*.md` остаются в репозитории как **исходник правды** — при
изменении их нужно вручную перенести в соответствующие Code-ноды hr_generate.

**Вызовы Ollama внутри hr_generate** идут через HTTP Request к `http://127.0.0.1:11434/api/generate`
(не через langchain-ноду — она доступна только в основном агенте hr_agent).

**Маршрутизация сценариев:** агент в системном промпте присваивает `scenario = "progul_ochny"` только
для очного прогула (сотрудник не ходит в офис), во всех остальных случаях — `"zaglushka"`.

## Настройка и запуск

### Запуск окружения (вручную)
```bash
brew services start postgresql@16   # обычно уже авто-старт (БД пока не задействована)
brew services start ollama          # обычно уже авто-старт
n8n start                           # http://localhost:5678
```

### 1. Создать Credential: Ollama API
n8n → Settings → Credentials → Add → Ollama API
- Base URL: `http://127.0.0.1:11434`
> ⚠️ Именно `127.0.0.1`, не `localhost` — macOS резолвит в IPv6 (::1), Ollama слушает только IPv4

### 2. Импортировать workflow (вручную через UI n8n)
1. Импортировать `hr_generate.json` (Import from File)
2. Импортировать `hr_agent.json` → проверить credential Ollama (`Ollama account`)
3. В ноде `Генерация документов` (hr_agent) убедиться, что `workflowId` указывает на импортированный hr_generate

> Workflow подгружаются в n8n вручную через интерфейс. Скрипта автозапуска нет.
> Credential PostgreSQL для текущей архитектуры **не требуется** (сроки считаются в JS).

### 3. Тест
Открой `HR-агент` → Test Workflow → в чате: `"Помоги оформить прогул"`

## Как добавить новый сценарий

1. Создать `knowledge/{scenario}_checklist.md` — чеклист из соответствующего .docx
2. Создать `templates/sz_{scenario}.md` / `act_{scenario}.md` — если нужны другие шаблоны
3. В `hr_generate.json`:
   - добавить ветку в ноду `Выбор сценария` (Switch) под новый `scenario`-ключ
   - вшить чеклист/шаблон в соответствующие Code-ноды (промпты)
4. В системном промпте агента (`hr_agent.json`) описать, когда присваивать новый `scenario`-ключ

## Ключевые ограничения ассистента

- Никогда ничего не додумывает — только факты от руководителя
- Граница: до передачи пакета документов эксперту включительно
- Если ситуация не под сценарий → направить к Касмыниной О. или Черепановой Т.
- Если нарушение связано с ИБ / коммерческой тайной → юр. служба / ОИБО
- Не цитирует статьи ТК по памяти; сроки берёт только из `calc_deadlines`, не считает даты в уме

## Роли в процессе

| Роль | Описание |
|------|----------|
| Руководитель | Пользователь ассистента. Инициирует, предоставляет факты |
| ИИ-ассистент | Ведёт диалог, консультирует, формирует документы |
| Эксперт кадрового учёта | Черепанова Т., Касмынина О. Принимает финальный пакет |
| ОКУ | Получает табельные данные (НН), приказы |
