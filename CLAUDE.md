# HR-консультант — CLAUDE.md

## Что это

ИИ-ассистент для руководителей Контура по оформлению дисциплинарных документов (акт об отсутствии, служебная записка). Реализуется на **n8n** + **Claude (диалог)** + **Ollama (генерация)** + **Postgres** + **Google Drive (шаблоны)**. Статус: активная разработка, **архитектура v6 (Telegram-бот, рендер .docx, Postgres)**.

## Архитектура v6 — два workflow

Вся система собрана в **два workflow**: диалоговый агент (Claude) ведёт диалог в Telegram,
проверяет сроки и вызывает генерацию (Ollama + рендер .docx из шаблонов Google Drive).

```
HR-агент.json (точка входа)
    ↓
Telegram: входящие (Telegram Trigger, updates=message)
    ↓
Нормализация входа TG (Code: text / photo + валидация фото) ──┐
    ↓                                                          ↓
Есть фото? (IF) → Postgres: запись доказательства (hr_evidence)
    ↓
HR-ассистент (AI Agent, Claude Opus 4.8 через Anthropic Chat Model)
    ├── Postgres память (Postgres Chat Memory, sessionKey = chat_id, окно 30)
    ├── Tool: calc_deadlines («Рассчитать сроки ст.», toolCode/JS) — ПРЯМОЙ тул
    │         → детерминированный расчёт дедлайнов ст.193, без LLM-арифметики
    └── Tool: generate_documents → Генерация документов.json (toolWorkflow)
    ↓
Telegram: ответ (Send Message, parse_mode=HTML)
```

**Поток диалога (системный промпт агента):** приветствие → сбор фактов по одному вопросу →
суммаризация → **Шаг 4: проверка сроков** (обязательно ДО генерации) → подтверждение
руководителем → генерация → выдача документами в Telegram.

**Проверка сроков (Шаг 4).** Агент сам вызывает инструмент `calc_deadlines` (Code-нода на JS),
передавая строку `"violationDate=ГГГГ-ММ-ДД; discoveryDate=ГГГГ-ММ-ДД"`. Тот возвращает JSON:
`{ isExpired, expiredReason, violationDeadline, discoveryDeadline }`. Агент НЕ считает даты сам —
доверяет только числам из тула. Если срок истёк — генерация не запускается, руководителя направляют
к Касмыниной О. / Черепановой Т. **Сроки ст.193 проверяются ТОЛЬКО здесь, в main-агенте** — в
генерации они НЕ дублируются.

> ⚠️ **Не вкладывать саб-агентов!** В v4 проверку сроков делал отдельный AI Agent (`agentTool`
> «Ресерчёр по срокам»). Вложенный агент в n8n падает с ошибкой *«The Tool attempted to return an
> engine request, which is not supported in Agents»*. `calc_deadlines` висит прямым тулом.
> Логику саб-агентов всегда переносить на `toolCode`/`toolWorkflow`/HTTP Request Tool.

> **Буферные сроки** (внутренний буфер Контура, НЕ дословно ТК РФ; baseline ТК — 6 мес / 30 дн):
> - от даты нарушения: **+5 мес. 15 дн.**
> - от даты обнаружения: **+20 дн.**
> Зашиты прямо в JS-код `calc_deadlines`. Уточнять у Касмыниной О.

**Генерация документов.json — конвейер генерации (без проверки сроков; Ollama + рендер .docx):**

```
Вызов из агента (Execute Workflow Trigger)
  → Нормализовать данные (Code: распаковка query, chat_id, scenario)
  → Выбор сценария (Switch по scenario)
        ├ progul_ochny → Drive: скачать чеклист (Google Drive, Google Doc → text/plain)
        │                 → Прочитать данные прогула (Code: чеклист из binary)
        └ zaglushka    → Сценарий: заглушка
             ↓
        Vision: анализ доказательств (заглушка, Code)
        → Промпт: нарушения (RAG, Code) → [Статус: анализ нарушения ⏳] + Ollama: нарушения (HTTP)
        → Объединить нарушения (Code, прокидывает chat_id)
        → Значения документов (Code: context → плейсхолдеры + DOC_MAP флаги)
        → Ollama: умные значения (HTTP, JSON: падеж ФИО, обстоятельства, приложения)
        → Объединить значения (Code: мёрдж детерминированных + умных значений)
        → Статус: готовлю документы ⏳
        → Нужен акт? (IF по флагу нужен_акт)
              ├ да  → Drive: скачать акт → Рендер акта (adm-zip) → Telegram: акт (sendDocument)
              └ нет → дальше
        → Нужна СЗ? (IF по флагу нужна_сз)
              ├ да  → Drive: скачать СЗ → Рендер СЗ (adm-zip) → Telegram: СЗ (sendDocument)
              └ нет → дальше
        → Финальный ответ (Code, краткий текст) → Postgres: запись кейса (hr_cases) → Вернуть результат
```

**Рендер .docx (гибрид — почему так):**
- Шаблоны лежат в **Google Drive** как `.docx`/Google Docs с цельными `{{плейсхолдерами}}`.
  Команда правит их прямо в Drive — workflow трогать не нужно.
- **Service Account** (`N8N-kontur`) **скачивает** шаблон (download). Копировать/создавать файлы
  он НЕ может — у Service Account нет квоты хранилища (403 storageQuotaExceeded). Поэтому шаблон
  скачивается, заполняется **локально**, в Drive ничего не пишется.
- **Гибрид заполнения:** детерминированные поля (даты, ФИО, должности) Code собирает из `context`;
  «умные» поля (ФИО в дательном падеже, абзац обстоятельств, список приложений) даёт Ollama
  одним JSON-ответом (`format=json`). Оба набора мёрджатся в один плоский объект значений.
- **Подстановка** — Code-нода на `adm-zip`: `.docx` = zip, в `word/document.xml` меняем
  `{{ключ}}` → значение, пакуем обратно, отдаём бинарём в Telegram `sendDocument`.

> ⚠️ Рендер требует запуска n8n с переменной `NODE_FUNCTION_ALLOW_EXTERNAL=adm-zip`,
> иначе Code-нода не сможет подключить `adm-zip`.

**Вариативность документов (DOC_MAP).** В ноде «Значения документов» есть карта
`DOC_MAP = { scenario: { нужен_акт: bool, нужна_сз: bool } }`. Развилки `Нужен акт?` / `Нужна СЗ?`
включают/выключают рендер каждого документа по флагам. Это задел под новые сценарии (не у всех
нужен и акт, и СЗ).

**Чеклист — динамический, ст.193 — нет.** Чеклист прогула подгружается из Google Drive
(нода «Drive: скачать чеклист», Google Doc → text/plain). Текст ст.193 в генерации НЕ хранится —
сроки целиком на `calc_deadlines` в main-агенте.

**Вызовы Ollama внутри генерации** идут через HTTP Request к `http://127.0.0.1:11434/api/generate`
(langchain-модель используется только в диалоговом агенте, и там это Claude).

**Маршрутизация сценариев:** агент в системном промпте присваивает `scenario = "progul_ochny"` только
для очного прогула (сотрудник не ходит в офис), во всех остальных случаях — `"zaglushka"`.

**chat_id и Telegram.** Telegram Trigger требует публичный HTTPS-webhook → поднимается через
`cloudflared tunnel --url http://localhost:5678`, n8n запускается с `WEBHOOK_URL=<туннель>`.
URL туннеля меняется при каждом перезапуске — после смены нужно переактивировать workflow.
`chat_id` прокидывается в генерацию детерминированно (через `query`) и далее по цепочке нод —
по нему шлются промежуточные статусы ⏳ и готовые документы.

## Настройка и запуск

### Запуск окружения (вручную)
```bash
brew services start ollama                                  # генерация (qwen2.5:7b)
brew services start postgresql@16                           # память диалога + кейсы
cloudflared tunnel --url http://localhost:5678              # публичный webhook для Telegram
NODE_FUNCTION_ALLOW_EXTERNAL=adm-zip WEBHOOK_URL=https://<туннель> n8n start
```
> URL туннеля меняется при каждом перезапуске cloudflared — после смены переактивировать workflow
> (Active off/on), чтобы Telegram-webhook перерегистрировался на новый адрес.

### 1. Credentials в n8n

**Anthropic API** (диалоговый агент) — Settings → Credentials → Anthropic
- API Key: `sk-ant-...` (console.anthropic.com → API Keys; нужен пополненный баланс)
- Base URL: `https://api.anthropic.com` (ровно так, без `/v1`)

**Ollama API** (генерация документов) — Settings → Credentials → Ollama API
- Base URL: `http://127.0.0.1:11434`
> ⚠️ Именно `127.0.0.1`, не `localhost` — macOS резолвит в IPv6 (::1), Ollama слушает только IPv4

**Telegram API** — Settings → Credentials → Telegram. Токен от @BotFather.

**Postgres** — Settings → Credentials → Postgres. `host=localhost port=5432 dbname=hr_assistant user=olegkluev`.

**Google Drive (Service Account)** — credential `N8N-kontur`. Только чтение (download/export).
Файлы шаблонов и чеклиста должны быть расшарены на e-mail Service Account.

### 2. Применить схему БД
```bash
psql hr_assistant -f db/schema.sql        # таблицы hr_cases, hr_evidence
```
Таблицу `n8n_chat_histories` создаёт сама нода Postgres Chat Memory.

### 3. Импортировать workflow (вручную через UI n8n)
1. Импортировать `Генерация документов.json` (Import from File) → проверить credentials
   (Ollama, Telegram, Postgres, Google Drive `N8N-kontur`).
2. Импортировать `HR-агент.json` → проверить credential Anthropic на ноде `Anthropic Chat Model`,
   Telegram и Postgres.
3. В ноде `generate_documents` (HR-агент) убедиться, что `workflowId` указывает на импортированную
   «Генерацию документов».

### 4. Тест
Активируй оба workflow → напиши боту в Telegram: `"Помоги оформить прогул"`.
Изолированно отладить рендер .docx можно через `Тест-docx-рендер.json` (Execute Workflow).

## Как добавить новый сценарий

1. Создать `knowledge/{scenario}_checklist.md` — чеклист из соответствующего .docx; выгрузить
   его в Google Drive, прописать fileId в новую/существующую ноду скачивания чеклиста.
2. Подготовить `.docx`-шаблоны (акт/СЗ) с `{{плейсхолдерами}}` в Google Drive, прописать их fileId
   в ноде «Значения документов».
3. В `Генерация документов.json`:
   - добавить ветку в ноду `Выбор сценария` (Switch) под новый `scenario`-ключ;
   - в `DOC_MAP` (нода «Значения документов») добавить строку `scenario: { нужен_акт, нужна_сз }`.
     Развилки `Нужен акт?` / `Нужна СЗ?` сами включат/выключат рендер.
4. В системном промпте агента (`HR-агент.json`) описать, когда присваивать новый `scenario`-ключ.

## Ключевые ограничения ассистента

- Никогда ничего не додумывает — только факты от руководителя
- Граница: до передачи пакета документов эксперту включительно
- Если ситуация не под сценарий → направить к Касмыниной О. или Черепановой Т.
- Если нарушение связано с ИБ / коммерческой тайной → юр. служба / ОИБО
- Не цитирует статьи ТК по памяти; сроки берёт только из `calc_deadlines`, не считает даты в уме

## Роли в процессе

| Роль | Описание |
|------|----------|
| Руководитель | Пользователь ассистента. Инициирует, предоставляет факты |
