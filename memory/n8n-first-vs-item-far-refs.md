В n8n ноды **Telegram** и **HTTP Request** роняют входные данные — возвращают только свой ответ
(Telegram: `message_id, from, chat…`; Ollama HTTP: `response, context…`). Всё, что стоит ПОСЛЕ
такой ноды и обращается к прежним данным ссылкой `$('Нода')`, должно использовать **`.first()`**,
а НЕ **`.item`**.

- `.item` — «парный» элемент (paired item), привязан к текущему элементу через цепочку. **Рвётся**,
  когда поток разорван промежуточной нодой (Telegram/HTTP посередине) и при поштучном `Execute step`.
- `.first()` — берёт первый элемент выхода ноды напрямую, без привязки. Надёжен при полном прогоне
  (в этих workflow всегда 1 item на ветку).

**Симптомы потери:** развилки IF (`Нужен акт?` / `Нужна СЗ?`) улетают в False Branch; Telegram-нода
падает `Bad Request: chat_id is empty`; плейсхолдеры/флаги приходят `undefined`.

**Где всплыло (HR-консультант, июнь 2026):** в «Генерация документов» нода `Статус: готовлю документы`
(Telegram) стоит В ОСНОВНОМ потоке между `Объединить значения` и развилкой `Нужен акт?`. Она
роняла контекст → весь хвост (IF-развилки, Drive `*_file_id`, Postgres, рендеры) читал данные через
`$('Объединить значения').item` и `.item` рвался. Исправлено: **все** far-ссылки `.item.json` →
`.first().json` (16 шт. в двух workflow); chat_id потребители берут из одного стабильного места —
`Нормализовать данные` через `.first()`.

**Правила:**
1. После Telegram/HTTP-ноды читай прежние данные только через `.first()`.
2. Не гоняй критичные значения (chat_id) эстафетой через HTTP-ноды — бери из одной ранней стабильной
   ноды.
3. `$input.item.binary` (доступ к бинарю текущего item в рендере) — НЕ far-ссылка, его не трогать.
4. **Отлаживай только полным прогоном** (через бота). Поштучный `Execute step` рвёт far-ссылки
   by design — будет ложно показывать «потери».
