## Требования по блокам

### A. Канал взаимодействия → Mattermost (P0)

- Бот живёт в **Mattermost** (внутренний мессенджер Контура). Руководитель работает там.
- Telegram — **убрать** из целевого решения (можно оставить как dev-стенд).
- ⚠️ Следствие: хостинг должен достучаться до внутреннего контура Контура (сейчас прод в
  Нидерландах ради Anthropic — конфликт локаций, см. открытые вопросы).

### B. Идентификация через Стафф (P0, полная интеграция — решение Олега)

- **Прогульщик:** руководитель присылает ссылку на Стафф / логин → тянем ФИО, табельный номер,
  подразделение, должность, офис. Подтверждать корректность (данные на Стаффе бывают
  рассинхронизированы — напр. переехал офис, не актуализировано).
- **Свидетели:** тоже по ссылке на Стафф → должность, подразделение; в идеале проверять, что в
  одном офисе/юрлице.
- **Авторизация руководителя** — по аналогии с решением Максима: забирать данные по тому, кто
  пришёл (автор СЗ).
- **Юрлицо** — тоже желательно тянуть (свидетели должны быть в одном юрлице).
- 🔴 **Риск объёма (моё мнение, Клюев решил брать целиком):** это не одна интеграция, а связка
  (профиль + дерево подразделения + статусы + авторизация). **Больничных и командировок в Стаффе
  НЕТ** (Участник 1) — «чист ли свидетель в этот день» через Стафф не проверить, ручная сверка
  остаётся. Кадровичка советовала: ручной ввод + подтверждение сейчас, интеграция — на будущее.
  Кандидат на фазирование.

### C. Чеклист обязательных действий — в начало

- Если руководитель не знает процедуру — **сначала ознакомить с чеклистом** и провести по
  обязательным действиям, ПОТОМ собирать фактуру под документы.
- Обязательные действия из чеклиста: связаться по каналам связи; проверить в отделе кадров
  (больничный / отпуск / не оформлен ли); и т.д.
- Идея: **выдавать чек-лист как чек-бокс** — руководитель отмечает «не на больничном, не в
  отпуске, звонил, писал». Сначала прохождение чеклиста → потом кейс.
- В текущем чеклисте эта информация **уже есть** (подтвердил Участник 2) — перенести её на ранний этап.

### D. Сбор фактуры — правки вопросов

1. **Модуль умной проверки дат** — «сегодня / вчера / послезавтра» → конкретные даты. Прогул
   обычно в прошлом (вчера), не в будущем.
2. **Время отсутствия:** может быть полдня / опоздание, не только целый день. Спрашивать «с
   какого времени отсутствует», «до сих пор отсутствует или уже нет».
3. **«Отсутствовал» vs «отсутствует»** — учитывать прошедшее/настоящее (период «с X по настоящее время»).
4. **Город + полный адрес** рабочего места — тянуть и подтверждать (вплоть до города).
5. **Канал связи** — спрашивать, по какому каналу связывались (телефон / мессенджер). Важно,
   т.к. к СЗ прикладывается скриншот переписки.
6. **Как сотрудник аргументирует причину** — спрашивать прямо.
7. **Документальное подтверждение** (больничный / справка) — из чеклиста. Нет подтверждения →
   неуважительная причина. Доп.вопрос: обращался ли сотрудник за медпомощью.
8. **Время работы по трудовому договору** — НОВЫЙ вопрос к руководителю (с 9 до 18, с 8 до 17 и
   варианты). Если «не знаю» → направить в отдел кадров (заявка **911** / **9 кабинет** —
   уточнить). ⚠️ Бывает в ТД «сам определяет режим» — тогда акт не поставить.
9. **Последствия отсутствия** — НОВЫЙ обязательный вопрос для СЗ: к чему привело / приведёт
   отсутствие. **Предложить варианты** (как чеклист): упущенные клиенты, перераспределение
   нагрузки на других, невыполнение плановых показателей, негативное влияние на дисциплину
   команды. Формулировать и «привело», и «приведёт» (настоящее + будущее).
10. **Подробный рассказ + валидация:** просить руководителя обогатить кейс подробностями
    (звонил утром — не брал — скрин — вышел в 12:20 — скрин — объяснил, что проспал — к врачу не
    обращался). Додумывать за руководителя НЕ хотим, но просить детализировать — да.

### E. Проверка сроков ст.193 — ДВЕ раздельные проверки

- **Проверка для акта:** акт об отсутствии составляется в **день обнаружения**. Задним числом
  нельзя — это подделка документов. Если сотрудник уже вышел и отработал день — акт «вчерашним»
  числом не оформить.
- **Проверка для СЗ:** сроки привлечения (по нарушению / по обнаружению).
- **Связка со сценарием:** прогул → акт + СЗ. **Дистант → только СЗ** (акта нет — это снимает
  вопрос «может ли быть СЗ без акта»).
- Для фиксации важнее **срок обнаружения**. Проверка не проходит, если срабатывает хоть одна.
- **Вывод в позитивном сценарии:** убрать конкретные даты крайних сроков, писать просто «срок не
  истёк, можно привлекать». Не вываливать «крайние сроки по нарушению 09.12.2026…».
- **Вывод в негативном сценарии:** расшифровать по-человечески («по нарушению прошло больше
  5,5 месяцев»), даты тоже лучше убрать/расшифровать.
- Озвученный участниками baseline: **6 месяцев / 1 месяц** + «7 дней на рассмотрение». ⚠️
  Расходится с текущим буфером в калькуляторе (5 мес 15 дн / 20 дн) — уточнить у Касмыниной.

### F. Генерация документов — правки шаблонов

**Акт:**
- **Время вместо даты в конце.** Указывать **время окончания нарушения**, не дату:
  - прогул целый день → время **окончания рабочего дня** (фиксируем в конце дня);
  - опоздание → **время прихода** сотрудника (пришёл 10:07 — акт в 10:10, не спустя 10 часов).
- **Подразделение — всё дерево отдела** («в помещении [отдел/сектор]»). Тянуть полностью,
  **кроме департамента** (у департамента список исключений — пока игнорируем).
- **Город обязателен** (в текущем шаблоне города не было) — полный адрес вплоть до города.

**Служебная записка:**
- **«отсутствовал» (1 день, такого-то числа) vs «отсутствует по настоящее время»** (период:
  пропал и не вышел — «с X по настоящее время», акты кооперируются).
- **Больше конкретики** в изложении (детальный рассказ руководителя + скриншоты по ходу).
- **Раздел последствий** (см. D.9).
- 🐞 Баг отладки: СЗ тащит **все скриншоты подряд** (не те, что нужно) — поправить.

### G. Приоритизация сценариев

- **Запускаемся с ТОП-3** (самые частые): отсутствие на дистанте, разовый прогул, опоздание.
- Остальное сложнее — вряд ли легко отладим.
- **Одно «сложное» на пробу:** неисполнение должностных обязанностей (НДО) — нарушение
  норматива / невыполнение задачи. Если выйдет сыро — зарежем/отложим.
- Адаптация/наставничество — это больше про **испытательный срок**, отдельная история.

### H. Прочее

- **Статусная панель** «кто зашёл / какой шаг» — НЕ нужна. Только выход (документ). Олег
  оставляет для отладки, потом закрывает.
- **Авторизация** — как у Максима (данные по пришедшему руководителю).
